  WHAT'S IMPLEMENTED
================================================================================

CONCEPT: Object Ownership on Sui
SOURCE: https://docs.sui.io/guides/developer/sui-101/object-ownership

TWO MAIN APPROACHES:
  1. FASTPATH OBJECTS (Address-Owned)
     • Single owner only (or immutable)
     • ~0.5 second finality (no consensus)
     • Lower gas costs
     • Perfect for: Personal assets, game items, wallets
     • Trade-off: Need custodian for multi-party coordination

  2. CONSENSUS OBJECTS (Shared)
     • Globally accessible via Mysticeti
     • ~1-2 second finality with atomic consistency
     • Higher gas costs (shared object overhead)
     • Perfect for: Smart contracts, game worlds, liquidity pools
     • Benefit: Fully trustless coordination via consensus

THREE ESCROW PATTERN IMPLEMENTATIONS:
  1. LOCK MODULE (both patterns)
     • Tamper-proof locking primitive
     • Key ID prevents silent modifications
     
  2. FASTPATH ESCROW (3 phases)
     • Lock → Register → Swap (with trusted custodian)
     • Safety: Key ID matching prevents tampering
     
  3. CONSENSUS ESCROW (2 phases)
     • Create shared escrow → Complete swap (no custodian)
     • Safety: Move code + Mysticeti atomicity

FIVE OBJECT OWNERSHIP MODELS:
  ✓ Owned (single address)
  ✓ Shared (global access)
  ✓ Immutable (read-only)
  ✓ Wrapped (encapsulated)
  ✓ Dynamic Fields (key-value storage)

================================================================================
  DECISION FRAMEWORK
================================================================================

CHOOSING BETWEEN FASTPATH AND CONSENSUS:

  Q1: Need multi-party coordination?
      → YES: Go to Q2
      → NO: Use Fastpath (simpler, faster)

  Q2: Have trusted custodian?
      → YES: Use Fastpath Escrow (better latency)
      → NO: Use Consensus Escrow (fully trustless)

  Q3: Is latency <1 second critical?
      → YES: Fastpath + accept custodian requirement
      → NO: Consensus (trustless) is acceptable

  Q4: Need atomic multi-party updates?
      → YES: Consensus Escrow (only option)
      → NO: Fastpath Escrow (simpler protocol)

USE FASTPATH WHEN:
  ✓ Minimizing latency is critical
  ✓ Objects are personal/single-owner only
  ✓ You have trusted infrastructure
  ✓ Gas optimization is priority
  ✓ No multi-party coordination needed
  
USE CONSENSUS WHEN:
  ✓ Multiple parties need simultaneous access
  ✓ Trust-minimized model required
  ✓ Objects have shared state
  ✓ Atomicity is critical
  ✓ Audit trail via consensus needed

================================================================================
  CROZZ ECOSYSTEM APPLICATION
================================================================================

RECOMMENDED PATTERNS:

  Game Items ............... Fastpath (owned)
    └─ Personal NFTs, no sharing needed, fast transfers

  Game World ............... Consensus (shared)
    └─ Multi-player state, atomic updates required

  Player Accounts .......... Fastpath (owned)
    └─ Personal progression, single-owner data

  Token State .............. Consensus (shared)
    └─ Protocol asset, requires all-party access

  Player Trades ............ Consensus Escrow
    └─ Trustless peer-to-peer, no custodian needed

  NPC Trades ............... Fastpath Escrow
    └─ Contract-as-custodian, lower latency for UX

  Liquidity Pools .......... Consensus (shared)
    └─ Concurrent swaps with atomic consistency
